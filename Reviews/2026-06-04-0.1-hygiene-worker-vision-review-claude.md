## Non-Blocking Recommendations

1. **`CHANGELOG.md` heading on tag.** The entry is currently `0.1.0 - Pending`. Flip to a dated heading
   (`0.1.0 - 2026-06-04` or similar) in the same commit that creates the tag so the changelog and the tag
   agree. Keep a `## Unreleased` placeholder above it for future work.
2. **Dependabot ecosystem coverage.** `.github/dependabot.yml` only watches `github-actions`. AnvilRunner has
   no third-party Swift package dependencies today, so this is correct now. If/when SwiftPM dependencies are
   introduced (e.g. swift-argument-parser in 0.2), add a `swift` ecosystem entry in the same PR — keeping the
   policy file honest avoids it silently going stale.
3. **`Scripts/enforce-local.sh` failure message.** The shim helpfully exits `2` with a clear pointer when
   `swiftanvil-enforcement` is missing, but it does not check for `swiftanvil-meta`. If the registry checkout
   is absent or stale, the downstream enforcement script will surface a less-readable error. Consider
   mirroring the `[ ! -d "$registry_root" ]` check with a matching message; cheap robustness.
4. **CI smoke step.** `swift run anvil-runner help` exercises the help path (verified at
   `Sources/AnvilRunnerCLI/main.swift:29`). Worth widening over time — at least exit-code checks for an
   unknown command (already prints "Unknown command:" at line 32) and `--help` aliasing. Not required for
   0.1, but it would catch CLI surface regressions cheaply.
5. **CODEOWNERS scope.** `* @v-i-s-h-a-l` is appropriate for a solo-maintained repo. Note that GitHub does
   not request a CODEOWNERS review on a PR authored by the owner, so this is informational only — no
   accidental self-block. When a second maintainer joins, splitting `Sources/AnvilRunner/Cleanup/**` (the
   safety-critical surface) to require an extra reviewer would be a high-leverage change.
6. **README sibling-layout assumption.** Both `README.md` and `AGENTS.md` now point at
   `Scripts/enforce-local.sh`, which requires `swiftanvil-enforcement` and `swiftanvil-meta` to be cloned as
   siblings of this repo. New external contributors will hit that error first. A one-line
   `CONTRIBUTING.md` (or a `Local Setup` subsection in README) describing the expected workspace layout
   would smooth onboarding. Non-blocking; vision doc already implicitly assumes this.
7. **Managed Worker Vision phasing.** The order (capability discovery → doctor → provisioning → profiles →
   fleet) is correct: facts before judgement, read-only before mutation, single-worker before fleet. One
   sharpening: capability discovery should explicitly commit to a stable JSON schema *before* `doctor`
   consumes it programmatically, so the schema is not retro-fitted from doctor's needs. Worth a sentence in
   Phase 2.
8. **`anvil` vs `anvil-runner` command surface.** The vision doc presents a future `anvil worker …` /
   `anvil runner …` shape while the current executable is `anvil-runner`. Calling out the renaming/wrapping
   path (separate `anvil` umbrella binary? alias? new executable target in the same package?) in Phase 1
   would prevent ambiguity when the umbrella command is introduced. Not urgent — Phase 1 is "keep the doc
   current" — but worth a one-line "decision pending" note.
9. **Stale `1.0.0` reference check.** README's SPM snippet is now `from: "0.1.0"`. Worth a quick grep for
   any remaining `1.0.0` in source/doc files before tagging; the verification log says this was already run,
   so this is just a reminder to repeat it on the release commit.

## Review Questions — Direct Answers

1. **Cohesive for users while allowing internal separation later?** Yes. The single user-facing
   `anvil …` command surface plus the explicit "no new public repos until a second consumer exists" rule
   in both `AGENTS.md` and `MANAGED_WORKER_VISION.md` is the right discipline. It avoids the common
   premature-microservices failure mode without painting the project into a monolith corner.
2. **0.1 release-hygiene additions sufficient?** Sufficient to tag `0.1.0` for a solo-maintainer SwiftAnvil
   package. Recommendations 1, 6, and 9 above are the items I would batch into the tag commit; everything
   else can land post-tag without harm.
3. **Phase order correct?** Yes. Capability discovery (facts) → doctor (read-only judgement on those facts)
   → provisioning (mutation with dry-run) → profiles (declarative spec over the now-stable primitives) →
   fleet (only after one worker is reliable). The non-goals list correctly forbids the typical shortcuts
   (silent host mutation, mandatory Tailscale/SSH, hidden privileged ops).
4. **Should any changes be split before PR?** No. The hygiene files, CI matrix expansion, README/AGENTS
   updates, ROADMAP edits, and Vision doc form one coherent "ready to cut 0.1.0" unit. Splitting the Vision
   doc would orphan the README link it adds. Keep them together.
5. **Missing org-standard practices that should block?** None blocking. Items worth adding within the next
   one or two PRs (not this one): a `CONTRIBUTING.md` with sibling-layout instructions, a Swift ecosystem
   entry in Dependabot once a real dependency is added, and a `## Unreleased` section in CHANGELOG to
   establish the running-log convention.
